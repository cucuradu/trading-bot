# ML Pipeline Architecture & Mechanics

This document provides a deep dive into how the `ml-insights.json` payload is generated by the local Ubuntu ML pipeline. It details the underlying mathematical models, data transformations, and defensive programming measures (like look-ahead bias mitigation and idempotency) used to generate the daily insights.

---

## 1. Pipeline Orchestration (`generate_insights.py`)
The pipeline runs daily via cron and acts as an assembly line. It executes six distinct phases:
1. **Data Pipeline:** Fetches OHLCV data from Yahoo Finance and engineers features.
2. **Regime Model:** Classifies the current market state (Bull, Caution, Neutral, Defensive).
3. **Volatility Model:** Forecasts annualized realized volatility via GARCH.
4. **Ranking Model:** Predicts forward returns for 11 sectors using XGBoost.
5. **Topology (GNN):** Calculates systemic fragility.
6. **Assembly:** Merges all outputs into the `ml-insights.json` contract.

### Defensive Mechanics
- **Idempotency:** The assembly script tracks daily runs in an append-only `history.jsonl` file. If a run occurs multiple times on the same date, it simply overwrites the JSON file and ensures no duplicated lines enter the JSONL history.
- **Look-Ahead Bias Mitigation:** When predicting the regime or future returns, the pipeline strictly shifts features by one day (`T-1`). The model evaluates the state at the close of `T-1` to output a prediction for day `T`, guaranteeing that future price action is never inadvertently leaked into the feature set.

---

## 2. Market Regime Classification (GMM + Persistence)

The regime model uses an unsupervised **Gaussian Mixture Model (GMM)** applied to SPY features.

- **Features Used:** Distance from 50-day and 200-day Simple Moving Averages, recent realized volatility, and VIX levels.
- **Cluster Mapping:** The GMM identifies 4 distinct clusters in the feature space. These clusters are dynamically mapped to `Bull`, `Caution`, `Neutral`, and `Defensive` based on their average annualized return and volatility profiles.
- **Persistence Filter:** To prevent rapid "whipsawing" between regimes on noisy days, the output runs through a persistence filter. A regime must hold continuously for `N` consecutive bars (usually 3) before the filter officially flips the pipeline's output regime. Until the streak is met, the pipeline remains in the prior regime.
- **Breadth Divergence:** A hard-coded heuristic compares the 20-day return of SPY (market-cap weighted) vs RSP (equal-weight). If SPY is positive but RSP is negative, breadth divergence is triggered. If the GMM output is `Bull`, this divergence automatically downgrades the regime to `Caution`.

---

## 3. Sector Ranking & Allocation (XGBoost)

The ranking engine evaluates 11 SPDR sector ETFs to find the most optimal places to deploy capital.

- **Model:** `xgb.XGBRegressor` tuned via Optuna.
- **Target Variable:** The 5-day forward return of the sector ETF.
- **Cross-Validation:** The pipeline strictly uses Walk-Forward `TimeSeriesSplit` (anchored) with a purge window. Standard K-Folds are strictly forbidden as they leak future data.
- **Regularization:** Financial data is incredibly noisy. To prevent the XGBoost trees from collapsing into a single generic prediction, strict L1 (`reg_alpha`) and L2 (`reg_lambda`) penalties are applied, forcing the tree to grow shallow but diverse branches.

### Portfolio Allocation
The allocation algorithm strictly respects the constraints dictated by the GMM Regime:
1. The regime maps to an allowed number of `trade_slots` (e.g., Bull = 3 slots, Caution = 1 slot, Defensive = 0 slots).
2. The pipeline dynamically slices the top `N` sectors from the XGBoost rankings where `N = trade_slots`.
3. If `trade_slots > 0`, the pipeline applies **Inverse Volatility Weighting**. Each chosen sector's weight is inversely proportional to its 21-day annualized realized volatility, equalizing the risk contribution of each slice of the portfolio. If `trade_slots = 0`, the `universe_weights` dictionary is returned empty.

---

## 4. Volatility Forecasting (GARCH)

The pipeline employs a **Generalized Autoregressive Conditional Heteroskedasticity (GARCH(1,1))** model to forecast near-term volatility.

- **Distribution:** Uses a Student's t-distribution to account for the "fat tails" commonly seen in financial returns.
- **Scaling Mechanics:** Because financial daily returns are incredibly small decimals, the optimizer often fails to converge. The pipeline multiplies all log returns by 100 before fitting the GARCH model. It then generates the variance forecast, square-roots it, annualizes it ($\times \sqrt{252}$), and outputs the raw percentage (e.g., `12.18`).
- **Term Structure:** The pipeline pulls spot VIX and 3-Month VIX (VIX3M) to calculate the term structure ratio, yielding either `contango` or `backwardation`.
